# AI-to-AI Messaging Metrics

This benchmark view measures what matters for AI-to-AI message streams: how
many verified request/response exchanges fit through a constrained link, what
the tail latency looks like, how many framed bytes each semantic exchange costs,
and how much CPU the codec spends to achieve that.

For AURA, "framed bytes" should be read as the bytes carried after the AIWire
structure handshake: static dictionary agreement, session templates, and any
template-update signals establish the shared structure; the hot path should move
compact changes against that state rather than whole JSON-shaped frames.

The current measurement used the bidirectional TCP stress harness with a modeled
10 Mbps egress link in each direction, 2 ms one-way delay, 1 ms jitter, and a
pipeline window of 16. The benchmark corpus is generated by
`build_ai_wire_messages()` and contains protocol-shaped AI traffic: JSON-RPC,
tool calls, agent traces, handoffs, reviews, and responses.

## Metrics

- `Completed`: verified request/response exchanges completed before the 5 second
  deadline.
- `Ex/s`: semantic exchanges per second, not compressor calls per second.
- `Framed B/ex`: bytes carried by the link per completed exchange, including the
  harness frame length prefixes. For AIWire this is the session delta carried
  after structural negotiation, not the conceptual whole message.
- `BW cap ex/s`: bandwidth-proportional exchange capacity. It is calculated per
  direction as `link bytes per second / framed bytes per exchange`, then the
  lower request/response direction is used as the bottleneck capacity.
- `BW gain`: bandwidth-proportional exchange capacity versus raw JSON.
- `Obs/BW cap`: how much of the bandwidth-proportional capacity the observed
  run actually used. If this is low, bandwidth is no longer the bottleneck.
- `Saved`: framed wire bytes saved versus raw JSON for the same exchange stream.
- `p95` / `p99`: roundtrip latency tail for a full request/response exchange.
- `Codec CPU us/ex`: combined client and server encode/decode CPU time per
  exchange.
- `Semantic Mbps`: uncompressed AI message bytes delivered per second.
- `Link Mbps`: framed wire Mbps consumed by the codec on the modeled link.

## Readout

`aitoken_aiwire` is the bandwidth winner: it moves each verified AI exchange in
about 125 framed bytes and saves 89.3% versus raw JSON. Under a purely
bandwidth-bound model on these 10 Mbps directions, that gives about 12,948
exchanges per second, or 7.38x raw JSON capacity. That is the codec to prefer
when the link is the bottleneck and the peer session has already negotiated the
structure.

`aiwire` is the balanced winner in this run: it completed the most exchanges,
used the least non-raw codec CPU, and still saved 86.6% of framed wire bytes.
That is the default to prefer when CPU and bandwidth both matter.

The gap between `BW cap ex/s` and observed `Ex/s` is the next engineering target.
Raw JSON uses essentially all of its bandwidth capacity, so it is bandwidth
bound. `aitoken_aiwire` uses only 19.6% of its bandwidth-proportional capacity,
so after compression the bottleneck moves to codec CPU, JSON work, Python
scheduling, and harness overhead. Saving bandwidth makes room for more messages;
the runtime then has to be parallel/native enough to fill that room.

The next corpus improvement is to model long-lived agent sessions more directly:
one handshake, stable templates, periodic template-update signals, and many
small value/status/tool-result deltas.

`aitoken` alone is useful as a structural binary representation, but not enough
by itself for the smallest wire footprint. It cuts framed bytes by 70.0%, while
the combined `aitoken_aiwire` path cuts them by 89.3%.

Raw JSON is the wrong baseline for constrained AI-to-AI messaging. It consumes
more link than the modeled 10 Mbps direction can carry in this pipelined
exchange pattern, which is why its framed saved value is negative after frame
prefix overhead.

## Reproduce

Start the server:

```bash
PYTHONPATH=src python tools/stress_ai_wire_roundtrip_z6.py server \
  --host 127.0.0.1 \
  --port 8913 \
  --runs 5 \
  --link-mbps 10 \
  --one-way-delay-ms 2 \
  --jitter-ms 1
```

Run the client:

```bash
PYTHONPATH=src python tools/stress_ai_wire_roundtrip_z6.py client \
  --host 127.0.0.1 \
  --port 8913 \
  --seconds 5 \
  --exchanges 20000 \
  --codecs raw,zlib,aitoken,aiwire,aitoken_aiwire \
  --link-mbps 10 \
  --one-way-delay-ms 2 \
  --jitter-ms 1 \
  --pipeline-window 16 \
  --output /tmp/aura_roundtrip_raw_zlib_aitoken_aiwire_combo_cpp_10mbps.json
```

Generate the metrics table:

```bash
python tools/analyze_ai_wire_benchmark.py \
  /tmp/aura_roundtrip_raw_zlib_aitoken_aiwire_combo_cpp_10mbps.json \
  --output /tmp/aura_ai_to_ai_messaging_metrics_cpp_10mbps.md
```

## Research Notes

The bandwidth-proportional capacity model follows standard network throughput
practice: RFC 6349 defines bottleneck bandwidth, RTT, and bandwidth-delay
product as the core values for TCP throughput testing, and gives the basic
relationship `BDP = RTT x bottleneck bandwidth`. It also recommends measuring
both directions because asymmetric links have different bottlenecks.

RFC 5166 is a useful guardrail for interpreting these results: transport
mechanisms should be evaluated across tradeoffs such as throughput, delay, loss,
response behavior, and robustness rather than a single winning number.

RFC 7323 is relevant when moving this benchmark to faster or longer paths. It
defines TCP window scaling for large bandwidth-delay-product paths, which is how
real TCP stacks keep enough data in flight to use high-capacity links.

References:

- RFC 6349, Framework for TCP Throughput Testing:
  https://datatracker.ietf.org/doc/html/rfc6349
- RFC 5166, Metrics for the Evaluation of Congestion Control Mechanisms:
  https://www.rfc-editor.org/info/rfc5166/
- RFC 7323, TCP Extensions for High Performance:
  https://www.rfc-editor.org/rfc/rfc7323.html
